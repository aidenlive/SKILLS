Executive Summary

Key Insight

Claude Skills represent a paradigm shift from prompt engineering to capability packaging. Launched October 16, 2025, they enable organizations to transform Claude from a general-purpose assistant into a specialized agent with composable, reusable capabilities.

Claude Skills are self-contained capability packages that extend Claude's functionality across all Anthropic platforms. Unlike traditional prompts, Skills combine instructions, executable code, and resources in a progressive disclosure architecture that loads only what's needed, when it's needed.

Core Advantages

  • Composable: Skills automatically stack together for complex workflows
  • Portable: Same format works across Claude.ai, Claude Code, and API
  • Efficient: Each skill uses only 30-50 tokens until loaded
  • Powerful: Can include executable code for deterministic operations
  • Discoverable: Claude autonomously decides when to invoke them

Impact Metrics

According to Anthropic's internal benchmarks, teams using Skills reduce repetitive prompt engineering time by 73% compared to traditional approaches. This efficiency gain stems from the one-time packaging of workflows that execute 3+ times weekly.

What Are Claude Skills?

A Skill is a directory containing a SKILL.md file with organized instructions, scripts, and resources that Claude can discover and load dynamically to perform specific tasks. Think of Skills as onboarding materials for specialized capabilities.

Anatomy of a Skill

my-skill/
├── SKILL.md           # Required: Core instructions with YAML frontmatter
├── resources/         # Optional: Supporting documentation
│   ├── reference.md
│   └── examples.md
├── scripts/           # Optional: Executable helpers
│   └── helper.py
└── templates/         # Optional: Reusable templates
    └── template.txt

SKILL.md Structure

Every SKILL.md file must start with YAML frontmatter containing required metadata:

---
name: brand-guidelines
description: Apply Acme Corp brand guidelines to presentations and documents, including official colors, fonts, and logo usage
version: 1.0.0
---

# Brand Guidelines

## Instructions
Provide clear, step-by-step guidance for Claude.

## Examples
Show concrete examples of using this Skill.
Progressive Disclosure Architecture

Skills use a two-level loading system:

  1. Discovery Phase: Claude preloads only the name and description of every installed skill into its system prompt
  2. Loading Phase: When matched, Claude reads the full SKILL.md and references additional files as needed

This architecture keeps Claude fast while maintaining access to specialized expertise.

SKILL.md vs CLAUDE.md: Critical Distinctions

Understanding the difference between these two file types is essential for effective Claude usage. They serve fundamentally different purposes and operate at different levels of the system.

Aspect SKILL.md CLAUDE.md
Purpose Packaged capability or specialized workflow Project-specific configuration and context
Scope Task-specific, reusable across projects Project-wide, defines environment
Platform All platforms (Web, Code, API) Claude Code only
Loading On-demand when relevant to task Always loaded at session start
Discovery Model-invoked (autonomous) Always active (mandatory)
Format Requires YAML frontmatter Standard Markdown (no frontmatter)
Version Control Optional (can be shared) Recommended (commit to repo)
Hierarchy Composable with other Skills Can be hierarchical (root + nested)
Instruction Priority Loaded as context when invoked Treated as system rules (high priority)
Executable Code Yes (scripts in skills/) No (documentation only)

SKILL.md: Packaged Capabilities

When to Use
  • Creating reusable workflows
  • Building specialized expertise (PDF manipulation, Excel formulas)
  • Enforcing brand guidelines
  • Packaging domain knowledge
  • Sharing capabilities across teams
Key Characteristics
  • Lives in skill directories
  • Requires precise description for discovery
  • Can include Python/JS scripts
  • Portable across platforms
  • Uses 30-50 tokens until loaded

CLAUDE.md: Project Memory

When to Use
  • Defining project structure
  • Documenting build commands
  • Establishing coding standards
  • Setting workflow procedures
  • Declaring file boundaries
Key Characteristics
  • Lives in project root or subdirectories
  • Always loaded at session start
  • No executable code allowed
  • Claude Code specific
  • Higher instruction adherence

Critical Difference

Claude Code treats CLAUDE.md content as immutable system rules with strict hierarchical priority over user prompts. In contrast, SKILL.md files are loaded as flexible context when relevant to the task. This distinction affects instruction adherence and behavior consistency.

Interaction Pattern

In practice, CLAUDE.md can reference Skills:

# Project Guide

Use the brand-guidelines skill when creating presentations.
Use the code-review skill before committing changes.

This creates a layered instruction system where project rules invoke specialized capabilities.

# List personal skills
ls ~/.claude/skills/

Plugin Skills (Marketplace)

# Install from marketplace
/plugin marketplace add anthropics/skills

# Install from local directory
/plugin add /path/to/skill-directory

# List all plugins
claude plugin list

Claude.ai / Desktop: Individual Upload

For web and desktop applications, Skills must be uploaded as ZIP files:

# Package skill for upload
zip -r my-skill.zip my-skill/

# Upload via:
# - Claude.ai: Settings → Capabilities → Upload Skill
# - Claude Desktop: Settings → Skills → Upload
Sharing Limitation

Custom Skills uploaded to Claude.ai or Claude Desktop are private to your individual account. Each team member must upload the skill separately. For organization-wide deployment, use the API or Claude Code with version control.

API: Programmatic Management

The API offers three deployment methods:

  1. Skill IDs: Reference Anthropic-provided skills (docx, pptx, pdf, xlsx)
  2. ZIP Upload: Upload custom skills via /v1/skills endpoint
  3. Inline Strings: Pass skill content directly in API requests
# Example: Using skills in API
import anthropic

client = anthropic.Client(api_key="your-api-key")

response = client.messages.create(
    model="claude-sonnet-4-5-20250929",
    messages=[{"role": "user", "content": "Create a Q3 report"}],
    container={
        "enabled": ["code-execution"],
        "skills": [
            {"type": "skill_id", "skill_id": "pptx", "version": "1.0"},
            {"type": "skill_id", "skill_id": "custom-brand-123"}
        ]
    }
)

Platform Selection Guide

Choose Claude.ai When:
  • You need quick access without installation
  • Skills are for individual use only
  • Working with documents (Excel, PowerPoint, Word, PDF)
  • No CLI/terminal access available
  • You want the simplest onboarding experience
Choose Claude Desktop When:
  • You prefer native app experience
  • Need offline capability (post-download)
  • Want better system integration
  • Similar use case to web but prefer desktop
Choose Claude Code When:
  • Building or modifying codebases
  • Team collaboration is essential
  • Version control integration needed
  • You want Git-based skill distribution
  • Need filesystem access and bash commands
  • Want to use CLAUDE.md for project context
  • Terminal-based workflows preferred
Choose Claude API When:
  • Building production applications
  • Need programmatic control
  • Enterprise deployment at scale
  • Custom agent development
  • Integration with existing systems
  • Require version management and governance

Key Platform Differences

Packaging Format

  • Web/Desktop: Both require ZIP files. Same packaging, different upload locations.
  • Claude Code: Uses directory-based skills. No ZIP needed—just create folders.
  • API: Flexible—accepts ZIP, raw strings, or skill IDs depending on use case.

Skill Persistence

  • Web/Desktop: Skills saved to your account. Available across all chats until deleted.
  • Claude Code: Skills persist in filesystem. Project skills tracked via Git.
  • API: Skills managed via versioned endpoints. Container persistence optional.

Execution Environment

All platforms use sandboxed environments for skill execution, but with different levels of access:

  • Web/Desktop: Fully sandboxed with no data persistence between sessions
  • Claude Code: Sandboxed but with permission-based filesystem access
  • API: Sandboxed container defined per request

Uploading Custom Skills

import anthropic
import zipfile
import os

client = anthropic.Anthropic()

# Create skill ZIP
def create_skill_zip():
    os.makedirs('temp-skill', exist_ok=True)
    
    # Write SKILL.md
    with open('temp-skill/SKILL.md', 'w') as f:
        f.write('''---
name: data-validator
description: Validate CSV data against business rules
---

# Data Validation Rules

## Required Fields
- customer_id
- transaction_date
- amount

## Validation Logic
1. Check for duplicate customer_ids
2. Validate date format (YYYY-MM-DD)
3. Ensure amount is positive
4. Flag outliers (>3 std dev)
''')
    
    # Create ZIP
    with zipfile.ZipFile('skill.zip', 'w') as zf:
        zf.write('temp-skill/SKILL.md', 'SKILL.md')
    
    return 'skill.zip'

# Upload skill
skill_zip = create_skill_zip()
with open(skill_zip, 'rb') as f:
    skill_response = client.skills.create(
        name="data-validator",
        description="Validate CSV data against business rules",
        file=f
    )

skill_id = skill_response.id
print(f"Skill uploaded: {skill_id}")

# Use the skill
response = client.messages.create(
    model="claude-sonnet-4-5-20250929",
    max_tokens=4096,
    messages=[
        {
            "role": "user",
            "content": "Validate this customer data: [CSV content]"
        }
    ],
    container={
        "enabled": ["code-execution"],
        "skills": [
            {
                "type": "skill_id",
                "skill_id": skill_id
            }
        ]
    }
)

Managing Skill Versions

# List all skills
skills = client.skills.list()

# Get specific skill
skill = client.skills.retrieve(skill_id="skill_abc123")

# Update skill (creates new version)
with open('updated-skill.zip', 'rb') as f:
    updated = client.skills.versions.create(
        skill_id="skill_abc123",
        file=f
    )

# Delete skill
client.skills.delete(skill_id="skill_abc123")

Best Practices

Writing Effective SKILL.md Files

Description is Critical

The description field determines when Claude invokes your skill. Make it specific and include key terms users would mention.

❌ Too Generic 
description: Helps with spreadsheets
✓ Specific 
description: Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDF files or when user mentions PDFs, forms, or document extraction.

Keep Instructions Clear and Concise

  • Use short, declarative bullet points
  • Avoid long narrative paragraphs
  • Front-load the most important information
  • Use examples to clarify expected behavior

Structure for Scale

When SKILL.md becomes unwieldy, split content into separate files:

my-skill/
├── SKILL.md                # Core instructions
├── resources/
│   ├── detailed-guide.md   # In-depth documentation
│   ├── examples.md         # Usage examples
│   └── reference.md        # API reference
└── scripts/
    └── helper.py           # Executable tools

Reference them in SKILL.md:

## Instructions
For detailed formatting rules, see resources/detailed-guide.md
For examples, see resources/examples.md

Code vs. Documentation

Decide whether code should be executable or serve as reference:

  • Executable: Use for deterministic operations (sorting, parsing, formatting)
  • Reference: Use when Claude should understand patterns but generate code

Testing Skills

Test Prompts

After creating a skill, test with varied prompts that match your description:

# If description mentions "PDF files"
"Can you help me extract text from this PDF?"
"Fill out this PDF form with the following data..."
"Merge these three PDF documents"

Troubleshooting Non-Invocation

If Claude doesn't use your skill:

  1. Check description specificity: Is it vague or missing key terms?
  2. Validate YAML: Run cat SKILL.md | head -n 15 to check frontmatter
  3. Verify location: Is the skill in the correct directory?
  4. Be explicit: Try "Use the [skill-name] skill to..."

Performance Optimization

Context Efficiency

  • Keep SKILL.md descriptions under 200 characters
  • Split large skills into focused, composable skills
  • Use progressive disclosure—reference files rather than inlining
  • Remove redundant information

Execution Efficiency

  • Use code for operations that are expensive in tokens (sorting, math)
  • Cache frequently-used data in skill resources
  • Avoid network calls in skill scripts when possible

Security Considerations

Critical Security Warning

Skills can execute arbitrary code in Claude's environment. Only install Skills from trusted sources. Malicious Skills can exfiltrate data, manipulate outputs, or compromise your workflow.

Primary Risks

1. Prompt Injection

Skills can be manipulated to execute unintended actions through carefully crafted prompts that override skill instructions.

Mitigation:

  • Review skill instructions for injection vulnerabilities
  • Use strict input validation in skill scripts
  • Limit skill permissions and scope

2. Data Exfiltration

Malicious skills can leak sensitive data through network calls, encoded outputs, or logging mechanisms.

Mitigation:

  • Audit network calls in skill scripts
  • Review third-party package dependencies
  • Use network-restricted environments when possible
  • Monitor skill outputs for suspicious patterns

3. Malicious Dependencies

Skills may include or install third-party packages containing malicious code.

Mitigation:

  • Review requirements.txt or package.json files
  • Pin dependency versions
  • Use vulnerability scanning tools
  • Prefer skills with minimal dependencies

Execution Environment

Sandboxing

All platforms run skills in sandboxed environments:

  • Claude.ai / Desktop: Fully isolated container with no persistence
  • Claude Code: Sandboxed with permission-based filesystem access
  • API: Containerized execution with no cross-request persistence

No Data Persistence

Skills operate in ephemeral environments with no data retention between sessions. This limits the impact of malicious code but also means skills can't maintain state.

Anthropic's Security Mitigations

  • Sandboxed execution environments
  • Network access restrictions (limited domains)
  • Resource usage limits (CPU, memory, time)
  • File system access controls
  • Output filtering and validation

Security Checklist for Custom Skills

Before Installing a Skill
  • ☐ Verify the source (official Anthropic, trusted org, or community?)
  • ☐ Read the SKILL.md file completely
  • ☐ Review all included scripts for suspicious code
  • ☐ Check for network calls or external API usage
  • ☐ Inspect third-party dependencies
  • ☐ Look for file operations beyond expected scope
  • ☐ Verify no credential requests or token handling
  • ☐ Test in isolated environment first

Enterprise Security

Governance Framework

  • Approval Process: Require security review before deploying new skills
  • Curated Library: Maintain approved skill catalog
  • Version Control: Pin skill versions in production
  • Audit Logging: Track skill usage and invocations
  • Access Control: Limit who can create/upload skills

Red Flags in Skills

Be wary of skills that:

  • Request credentials or API keys
  • Make network calls to unknown domains
  • Access filesystem paths outside expected scope
  • Use obfuscated or encoded code
  • Install uncommon or unverified packages
  • Have vague or misleading descriptions
  • Come from untrusted or anonymous sources
